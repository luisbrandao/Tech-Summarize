# Tech-Summarize — Agent Notes

Third-party SillyTavern UI extension. Maintains a persistent, three-section memory of the
roleplay (Characters / Events & Plot / World & Lore), generated by an LLM and injected back into
the main prompt so long chats keep their continuity.

## What this extension does

- Each section has its own summarization prompt (defaults loaded from `default-prompt-*.txt`)
  and its own stored content.
- Summaries are stored on chat messages at `chat[idx].extra.memory` as
  `{ characters, body, lore }` (legacy chats may hold a plain string — treat it as `body`).
  The latest stored memory wins; `getLatestMemoryFromChat()` walks backwards to find it.
- The combined summary is rendered through the user-editable injection template
  (`{{summary_characters}}`, `{{summary_body}}`, `{{summary_lore}}`, `{{summary}}`) and injected
  via `setExtensionPrompt(MODULE_NAME, ...)` under the key `tech_summarize`. Other extensions
  (e.g. SillyTavern-Director) read that key — do not rename it.
- Exposes the `/summarize` slash command and `{{summary}}` / `{{summary_<section>}}` macros.

## Prompt builders

Selected by `settings().prompt_builder` (`prompt_builders` enum in `index.js`):

| Builder | How it works |
| --- | --- |
| Classic (`DEFAULT`) | `generateQuietPrompt()` — core builds the full prompt, summary request appended. Blocking. |
| Raw blocking / non-blocking | `getRawSummaryPrompt()` concatenates unsummarized messages as plain text, sent via `generateRaw()`. |
| Connection profile (`CONNECTION_PROFILE`) | Builds its own chat-completion message array and sends it through an independent connection profile via `ctx.ConnectionManagerRequestService.sendRequest()`. Non-blocking; can target a different model than the main reply. Fills the unsummarized window forward (incremental). |
| Connection profile, whole chat (`CONNECTION_PROFILE_CLASSIC`) | Same connection-profile machinery, but Classic-style: `buildSummaryMessages(..., wholeChat=true)` fills the most recent messages that fit (backward from the end) and re-summarizes the whole chat each run. Anchors the stored summary at the newest message kept. |

### Connection-profile builder (the important one)

- Profile/preset resolution: `resolveSummaryConnection()`. `"current"` means the app's active
  profile / the profile's own preset. Profiles come from
  `ctx.extensionSettings.connectionManager.profiles`.
- Preset override is done by temporarily reassigning `profile.preset` around the request and
  restoring it in `finally` (`sendSummaryRequest()`); core derives the preset name from the
  profile object.
- The prompt deliberately contains **no system prompt** from the main connection. The system
  message is assembled by `buildSummarySystemPrompt()` as labeled sections: character card,
  player persona, Author's Note (honoring its configured position; in-chat notes are spliced
  into history at depth by `buildSummaryMessages()`), active World Info (dry-run scan via
  `ctx.getWorldInfoPrompt(chat, maxContext, true)` — never the event-emitting variant), and the
  section's previous summary. The "No WI/AN" checkbox (`SkipWIAN`) drops both WI and the note.
- History fill has two modes (`buildSummaryMessages(..., wholeChat)`): the default **incremental**
  mode fills **forward from the first unsummarized message** (oldest first) — summarization consumes
  the oldest pending messages so the next run picks up where this one stopped; the **wholeChat**
  mode (the `CONNECTION_PROFILE_CLASSIC` builder) fills the most recent messages **backward from the
  end** and re-summarizes everything each run, like Classic. The anchor message index
  (`lastUsedIndex`) is where the new summary gets stored — the window end for incremental, the
  newest kept message for wholeChat.
- **Per-section windowing, one source of truth.** The three sections summarize independently. Each
  section's content is stored under its own key in `mes.extra.memory` (`{ characters?, body?, lore? }`),
  and a save writes **only the touched section's key** (`saveSectionToMessage(index, section)`) — never
  a full three-section snapshot — so one section's save can't clobber another's value or progress.
  - Both readers select the **same message**: the newest one that *defines* the section's key
    (`Object.prototype.hasOwnProperty`, empty included). `getLatestMemoryFromChat(chat)` returns that
    message's value (so a deliberate clear, stored as `''`, is honored); `getIndexOfLatestChatSummary
    (chat, section)` returns that message's index — the anchor the next Raw/Profile run windows from.
  Selecting on key *presence* (not non-emptiness) is what keeps content and window pointing at the
  same message, so they can never diverge (no separate anchor marker to drift). Because the next run
  windows from anchor+1, each summary lands strictly after the current anchor — the anchor only moves
  forward, so a cleared/empty value is never shadowed by an older one. The original bug was a single
  global marker: summarizing the first section planted an entry the other two read as "already
  summarized", so they found zero messages and errored. Manual edits, restores, and Classic saves all
  flow through `saveSectionToMessage` and re-anchor only the section they touch, consistently with
  their content. (A save on a one-message chat is skipped — the lone message is the one both readers
  drop, so it can't be read back.)
- The summarize instruction must end the array as a **user** turn. Chat-completion backends
  always append a generation prompt; a trailing assistant message renders as a completed turn
  and the model emits an end token after ~1 token.
- Token budgeting: `contextSize - responseTokens - CONTEXT_SAFETY_MARGIN`, counted with
  `getTokenCountAsync()` plus `PER_MESSAGE_OVERHEAD` per message.

## Architecture & conventions

- Single-file extension: all logic in `index.js`; UI in `settings.html` rendered through
  `renderExtensionTemplateAsync()`; styling in `style.css`. No build step, no dependencies.
- Settings live at `extension_settings.tech_summarize` (`SETTINGS_KEY`); always go through the
  `settings()` accessor and call `saveSettingsDebounced()` after mutating.
- `loadSettings()` backfills missing keys from `defaultSettings` and migrates legacy shapes —
  new settings need a default there and (if visible) UI sync in the same function.
- UI listeners are (re)bound in `setupListeners()` with `.off().on()` because the panel can be
  popped out (`doPopout()` clones the DOM); never bind listeners anywhere else.
- Section IDs follow `ts_section_<thing>_<section>` and are generated in loops over
  `summarySections` — keep new per-section UI in that pattern.
- Default prompts are plain-text files fetched at init relative to `import.meta.url`
  (`loadDefaultPrompts()`), so the repo can be renamed/moved without code changes.
- Guard everything read off `getContext()` — fields differ across SillyTavern versions; prefer
  optional chaining plus a `console.warn` fallback over letting init throw.
- After an async generation completes, check `isContextChanged(context)` before writing results;
  the user may have switched chats mid-request.

## Build / test / release

- No build step. Reload via SillyTavern → Extensions → "Reload UI" (or hard-refresh the page).
- Manual testing only: pick a chat with enough messages, force-summarize a single section, and
  verify the result lands both in the textarea and on `chat[idx].extra.memory`.
- Bump `version` in `manifest.json` on every user-facing change.

## Commit & PR expectations

- Imperative commit titles prefixed by type (`feat:`, `fix:`, `change:`), matching the existing
  history.
- For non-obvious fixes, explain the root cause in the commit body.
